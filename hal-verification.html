How Your DNS Pipeline Works

Before troubleshooting, it helps to understand the chain of events every DNS query takes on your network. Every time a device asks "what's the IP for ads.doubleclick.net?", this is the exact path:

Device → eero router → Pi-hole (192.168.4.2:53) → upstream DNS (1.1.1.1 or 8.8.8.8) If domain is on blocklist: Pi-hole returns 0.0.0.0 → Ad never loads If domain is NOT blocked: Pi-hole forwards upstream → Returns real IP

The diagnostic layers below match this pipeline. If anything in this chain is broken — eero not pointing to Pi-hole, Pi-hole container down, DNS bypassed by app — ads get through. Each layer builds on the previous one, so start at Layer 1 and work forward.

Why Am I Still Seeing Ads?

🔍 Start Here: Quick DNS Check

Run these two commands in your Mac Terminal right now:

nslookup google.com nslookup ads.doubleclick.net

What to look for:

  • The Server: line should show your Pi-hole's IP (e.g., 192.168.4.2). If it shows 1.1.1.1, 8.8.8.8, or anything else, DNS is bypassing Pi-hole entirely.
  • For ads.doubleclick.net: If Pi-hole is working, it should return 0.0.0.0 or NXDOMAIN. If it returns a real IP address, Pi-hole isn't blocking it.

The 5 Most Common Reasons Pi-hole Isn't Blocking Ads

1. ⚠️ eero Secure/Plus is overriding your DNS

The problem: eero has its own DNS filtering service. When you enable eero Secure or eero Plus with Advanced Security enabled, it intercepts all DNS queries network-wide and blocks the ones it knows about. Your custom DNS setting is effectively ignored.

Symptom: Pi-hole's Query Log shows 0 queries, or only a few from your Mac. Most devices don't appear in the log at all.

Fix: In the eero app, go to Menu → Network Settings → Advanced Settings → DNS, and check if Advanced Security is enabled. If so, toggle it OFF. You'll see a message like "Advanced Security has been disabled." Wait 2-3 minutes for the change to propagate to all devices.

2. 🔌 The device isn't using Pi-hole's DNS

The problem: Your device has a different DNS server configured, either from a hardcoded setting, an old DHCP lease, or because it's using DNS-over-HTTPS.

Symptom: nslookup on that specific device shows a different server than Pi-hole's IP. The Pi-hole query log doesn't show queries from that device.

Fix:

  • First, try refreshing the device's DHCP lease by toggling Wi-Fi off/on
  • Check the device's Settings → Wi-Fi → Advanced DNS settings and ensure it's set to DHCP (automatic) or explicitly to Pi-hole's IP
  • Some apps ignore system DNS and use hardcoded resolvers (Firefox, some game clients). See section on DNS-over-HTTPS below.

3. 🔴 Pi-hole container is down

The problem: The Docker container crashed, the Raspberry Pi rebooted, or the container doesn't have a restart policy.

Symptom: You can't reach the Pi-hole admin dashboard. SSH to the Pi and run docker ps | grep pihole — the container doesn't appear in the list.

Fix: SSH into your Pi and restart the container:

cd ~/homelab/pihole docker compose up -d

4. 📺 The ad isn't DNS-based (first-party ads)

The problem: Some ads are served from the same domain as the content itself. YouTube ads, Spotify ads, and Reddit's promoted posts all come from the content provider's own servers. Pi-hole cannot block these without breaking the service.

Symptom: You see ads on YouTube, Spotify, Reddit, or streaming services, and the Pi-hole query log shows the domain is NOT being blocked.

Fix: None. This is a limitation of DNS-based blocking. These services have decided that this is part of their business model, and it's not possible to block without a subscription (YouTube Premium, Spotify Premium, etc.).

5. 🔐 App is using DNS-over-HTTPS (DoH)

The problem: Firefox (and some other apps) send DNS queries encrypted directly to a hardcoded resolver, completely bypassing your network's DNS configuration.

Symptom: nslookup from your Mac shows Pi-hole, but Firefox is still seeing ads. The Pi-hole query log doesn't show any queries from Firefox.

Fix: In Firefox:

  1. Settings → Privacy & Security
  2. Scroll down to "DNS over HTTPS"
  3. Change from "Default Protection" to "Off"
  4. Restart Firefox

Chrome uses Secure DNS but respects your system's DNS configuration — no change needed.

Layer 1: Eero DNS Verification

Why start here?

The eero router is the DHCP server for your entire network — it's the one that hands out IP addresses AND tells every device which DNS server to use. If eero is pointing devices to 8.8.8.8 instead of your Pi-hole, no amount of Pi-hole configuration will help. Every device on your network asks eero where to send DNS queries on startup.

eero Secure/Plus is a special case: it intercepts all DNS queries at the router level, overriding your custom DNS setting entirely. eero custom DNS support docs →

Step 1: Confirm Custom DNS in eero

  1. Open the eero app on your phone
  2. Tap the Menu (hamburger icon)
  3. Go to Network Settings → Advanced Settings → DNS
  4. Confirm it shows "Custom DNS" enabled
  5. The primary DNS should show your Pi-hole's IP (default 192.168.4.2, or wherever you placed it)

Step 2: Check if eero Secure/Plus is interfering

CRITICAL: If you have an eero Secure or eero Plus subscription:

  1. In the eero app, look for Security or Advanced Security settings
  2. Check if it says "Advanced Security: ON" or similar
  3. If enabled, toggle it OFF. eero's own DNS filtering will override your custom DNS
  4. Confirm the message: "Advanced Security has been disabled" or similar
  5. Wait 2-3 minutes for the change to propagate
Eero DNS Verification Checklist
  • Custom DNS is enabled in eero
  • Primary DNS shows Pi-hole's IP
  • eero Secure/Plus Advanced Security is OFF
  • scutil --dns | grep nameserver on Mac shows Pi-hole IP
  • All devices have reconnected to Wi-Fi since the change

Step 3: Verify from Mac Terminal

scutil --dns # Look for nameserver entries

You should see your Pi-hole's IP listed under nameserver. If you see 8.8.8.8, 1.1.1.1, or other public DNS servers, the network DNS isn't pointing to Pi-hole.

Eero DNS not pointing to Pi-hole?
  • Double-check the IP address in eero DNS settings (typo?)
  • Disconnect and reconnect all devices to Wi-Fi
  • Toggle eero Secure OFF if you have it enabled
  • Restart the eero router (power off 30s, power on)
  • Restart Pi-hole container: docker compose down && docker compose up -d

Layer 2: Pi-hole Container Verification

Why check the container?

Pi-hole runs as a Docker container. If the container crashes (power outage, OOM kill, bad config), port 53 stops listening and every DNS query fails network-wide — no websites load for anyone. docker ps shows all running containers; if pihole isn't listed, nothing is answering DNS.

What you'll learn: How to read Docker container state, interpret startup logs, and verify a service is actually bound to a port. docker ps docs →  ·  docker logs docs →

Configuration Variables

Step 1: Check Container Status

SSH into your Raspberry Pi and verify Pi-hole is running:

ssh -l amervar@homelab.local docker ps | grep pihole

Expected output:

CONTAINER ID IMAGE STATUS PORTS abc123d pihole:latest Up 2 hours 0.0.0.0:53->53/udp, :::53->53/udp

The status should say Up X hours/minutes. If it says Exited or doesn't appear, the container is not running.

Step 2: Check Pi-hole Logs

docker logs pihole --tail 50

Look for: Any error messages or DNS startup logs. Pi-hole should log something like:

[services] pihole.service is enabled [DNS] Starting Pi-hole DNS server [DNS] Listening on port 53

Step 3: Verify Port 53 is Bound

DNS runs on port 53 (both TCP and UDP). Verify it's open:

sudo ss -tlnp | grep :53 sudo ss -ulnp | grep :53

Expected output (TCP): LISTEN ... 0.0.0.0:53 ... docker-proxy

Expected output (UDP): UNCONN ... 0.0.0.0:53 ... docker-proxy

Step 4: Access Pi-hole Admin Dashboard

Open your browser and navigate to:

https://192.168.4.2/admin

Important: Pi-hole v6 uses HTTPS and has a self-signed certificate. Your browser will show a warning:

  • Chrome/Edge: Click "Advanced" → "Proceed to [IP] (unsafe)"
  • Firefox: Click "Advanced..." → "Accept the Risk and Continue"
  • Safari: This warning cannot be bypassed in Safari. Use Chrome or Firefox
Pi-hole Container Verification
  • docker ps shows pihole container with status "Up"
  • Container logs show DNS server started successfully
  • Port 53 (TCP and UDP) is bound to docker-proxy
  • Admin dashboard accessible at https://[PIHOLE_IP]/admin
  • No recent container restarts (check logs for restart loops)
Pi-hole container not running?
  • Container not in docker ps: Run docker compose up -d from ~/homelab/pihole/
  • Container keeps restarting: Check logs with docker logs pihole --tail 100 for errors
  • Port 53 already in use: Check sudo ss -tlnp | grep :53. If systemd-resolved is listening, see pi-homelab-guide.html for fix
  • Can't reach admin dashboard: Verify container is running and check firewall rules with sudo ufw status

Layer 3: DNS Resolution Verification

Why test DNS resolution from your Mac?

Layers 1 and 2 confirm configuration. Layer 3 confirms actual behavior. nslookup and dig are DNS diagnostic tools that show you exactly which server answered a query and what it said. These are the same tools network engineers use to debug DNS at any scale.

Key concept: nslookup domain uses your system DNS (showing if Pi-hole is working end-to-end). dig domain @192.168.4.2 queries Pi-hole directly (useful for isolating whether Pi-hole itself has an issue vs. your Mac not using it).

nslookup manual →  ·  dig manual →

What's Your Mac Using for DNS?

Run this command in Terminal to see all DNS servers your Mac is configured to use:

scutil --dns

Look for: A section that lists your Pi-hole IP (e.g., nameserver[0]: 192.168.4.2). It should be the first entry.

Test DNS Resolution to Google

This should return a real IP address:

nslookup google.com

Expected output:

Server: 192.168.4.2 Address: 192.168.4.2#53 Non-authoritative answer: Name: google.com Address: 142.251.41.14

The Server line must show your Pi-hole's IP. If it shows 1.1.1.1, 8.8.8.8, or your ISP's DNS, queries are bypassing Pi-hole.

More Detailed Query with dig

For more verbose output, use dig and specify the Pi-hole IP directly:

dig google.com @192.168.4.2

This forces a query directly to your Pi-hole IP. Look for the ANSWER SECTION with the real IP.

Monitor Real-Time Queries in Pi-hole

Open the Pi-hole admin dashboard at https://192.168.4.2/admin and go to:

  1. Query Log tab
  2. Run nslookup google.com in Terminal again
  3. Watch the Query Log — it should show a new entry for google.com within seconds

If queries don't appear in the log, they're not reaching Pi-hole at all. Check Layer 1 (eero DNS) and Layer 2 (container status).

Flush Your Mac's DNS Cache

If you've recently changed DNS settings, flush the cache:

sudo killall -HUP mDNSResponder

Or simply toggle Wi-Fi off/on, which forces a fresh DHCP lease and picks up new DNS servers.

DNS Resolution Verification
  • scutil --dns shows Pi-hole IP as nameserver
  • nslookup google.com returns real IP from Pi-hole's server
  • dig query to Pi-hole directly works
  • Queries appear in Pi-hole Query Log in real time
  • DNS cache flushed on Mac after changes
DNS not resolving?
  • nslookup shows wrong server IP: Check eero DNS settings and flush Mac DNS cache
  • Queries don't appear in Pi-hole log: Device may be using a different DNS server. Check device settings directly
  • dig returns "connection refused": Pi-hole port 53 is not open. Check Layer 2 and verify firewall
  • DNS resolution times out: Ping Pi-hole IP first to verify network connectivity: ping 192.168.4.2

Layer 4: Ad Blocking Verification

Why test with known ad domains?

Layers 1–3 confirm the DNS pipeline is intact. Layer 4 confirms Pi-hole is actually blocking using its blocklists. Pi-hole uses DNS sinkholing: instead of forwarding a blocked domain to an upstream resolver, it responds immediately with 0.0.0.0 (a null IP). The device never makes a real connection to the ad server.

Testing with ads.doubleclick.net is reliable because it's on virtually every blocklist. If that domain returns a real IP, your blocklists are either disabled, empty, or Pi-hole isn't actually handling the query.

Pi-hole blocking modes explained →  ·  The Firebog — curated Pi-hole blocklists →

The Definitive Block Test

Run these commands in Terminal. If Pi-hole is working, the known ad domains should return 0.0.0.0 or NXDOMAIN:

# These SHOULD be blocked by Pi-hole nslookup ads.doubleclick.net nslookup pagead2.googlesyndication.com nslookup www.googleadservices.com nslookup ad.doubleclick.net nslookup static.ads-twitter.com
# These should return REAL IPs (not blocked) nslookup google.com nslookup apple.com nslookup github.com

Reading Pi-hole's Response

Response Meaning Status
0.0.0.0 or :: Domain blocked by Pi-hole ✓ Working
NXDOMAIN Non-existent domain (blocked) ✓ Working
Real IP address (e.g., 142.251.41.14) Domain not blocked ⚠ Not blocked (could be legitimate or not on blocklists)
Different server (1.1.1.1, 8.8.8.8, etc.) Pi-hole is not handling the query ✗ Pi-hole bypassed

Check Pi-hole's Query Log

For more details, check the Pi-hole admin dashboard:

  1. Open https://192.168.4.2/admin
  2. Go to Query Log
  3. Look for the domains you just queried
  4. Domains with a 🚫 icon (or red highlight) are blocked
  5. Domains with a ✓ (or green highlight) are allowed

⚠️ Important: First-Party Ads Cannot Be Blocked

If you see ads still appearing on YouTube, Spotify, Reddit, or streaming services, this is expected. These platforms serve ads from their own domains (first-party ads), and Pi-hole cannot block them without breaking the service. Pi-hole is most effective for:

  • Third-party banner ads on websites
  • Analytics and tracking domains
  • Ads on IoT devices, smart TVs, and apps
  • Blocking entire ad networks (DoubleClick, Google Ads, etc.)
Ad Blocking Verification
  • Known ad domains return 0.0.0.0 or NXDOMAIN
  • Legitimate domains return real IP addresses
  • Server line in nslookup output shows Pi-hole's IP
  • Query Log shows blocked domains with block indicators
  • Blocklists are active in Pi-hole dashboard (Adlists section)
Ad domains not being blocked?
  • Ad domains return real IPs: Check if blocklists are disabled in Pi-hole dashboard
  • Query log shows domain but it's not marked as blocked: The domain may not be on any of your active blocklists. Add more lists in Adlists section
  • Query doesn't appear in log at all: Query is not reaching Pi-hole. Check Layer 1-3
  • Still seeing ads on YouTube/Spotify: These are first-party ads. Pi-hole cannot block them. This is expected behavior

Layer 5: Traefik Reverse Proxy Verification

Why is Traefik separate from DNS?

Pi-hole handles which domains resolve. Traefik handles how HTTP/HTTPS requests are routed to services. They're complementary: Pi-hole blocks at the DNS layer before any connection is made, while Traefik routes legitimate traffic to the right container.

Traefik reads Docker labels on containers to discover routes automatically — no manual config file updates needed when you add new services. If Traefik is down, services still run but you can't reach them by hostname (only by direct IP:port).

Traefik official docs →  ·  Traefik Docker provider →

Check Traefik Container Status

SSH into your Pi and verify Traefik is running:

ssh -l amervar@homelab.local docker ps | grep traefik

Expected output: Container should show status Up and expose ports like 0.0.0.0:80->80/tcp, 0.0.0.0:443->443/tcp.

Check Traefik Logs

docker logs traefik --tail 30

Look for: Any startup errors or warnings. Traefik should show something like:

msg=Starting Traefik version=v3.x.x msg=Starting provider: docker

Verify Ports 80 and 443 Are Bound

Check that HTTP and HTTPS ports are open:

sudo ss -tlnp | grep :80 sudo ss -tlnp | grep :443

Expected output: LISTEN ... 0.0.0.0:80 ... docker-proxy and similar for 443.

Access Traefik Dashboard

Open your browser and navigate to:

http://192.168.4.100:8080

The Traefik dashboard should show:

  • HTTP Routers: All configured routes (including Pi-hole if you've added labels)
  • Services: All Docker services Traefik is aware of
  • Middlewares: Any auth or request modification rules

Check Pi-hole Routing (if configured)

If you've added Traefik labels to the Pi-hole container, it should appear in the dashboard. SSH to the Pi and check the docker-compose.yml:

cat ~/homelab/pihole/docker-compose.yml | grep -A5 labels

You should see labels like:

labels: - "traefik.enable=true" - "traefik.http.routers.pihole.rule=Host(\"pi-hole.local\")" - "traefik.http.services.pihole.loadbalancer.server.port=80"

Quick Traefik API Check

Get detailed JSON info about Traefik's configuration:

curl http://192.168.4.100:8080/api/rawdata | python3 -m json.tool | head -50
Traefik Verification
  • docker ps shows traefik container with status "Up"
  • Container logs show successful startup (no errors)
  • Ports 80 and 443 are bound to docker-proxy
  • Dashboard accessible at http://192.168.4.100:8080
  • Traefik can see Docker services (check HTTP Routers/Services tabs)
Traefik not working?
  • Container not running: Run docker compose up -d from the Traefik directory
  • Dashboard unreachable: Verify container is running and port 8080 is accessible
  • Services not appearing in dashboard: Check that Docker daemon is accessible in docker-compose.yml (volume mount to /var/run/docker.sock)
  • Port 80/443 in use: Check what's using them with sudo lsof -i :80
  • Routers not found: Verify labels are correctly formatted in service docker-compose.yml

What Pi-hole Cannot Block

Pi-hole is powerful, but it has limitations. Understanding what it can't do helps you set realistic expectations and avoid spending hours debugging what's actually expected behavior.

1. First-Party Ads

YouTube ads, Spotify ads, Reddit promoted posts, streaming service ads

These platforms serve ads from the same domain as the content itself. For example, youtube.com serves both content and ads, so Pi-hole cannot block one without breaking the other. This is intentional on their part — they've moved to first-party ads specifically to avoid DNS-based blocking.

Fix: Use premium subscriptions (YouTube Premium, Spotify Premium, etc.) or accept ads as part of the service.

2. DNS-over-HTTPS (DoH)

Firefox, some mobile apps, some game clients

Some applications send DNS queries encrypted directly to a hardcoded resolver (usually Cloudflare, Google, or Quad9), completely bypassing your network DNS. Pi-hole never even sees these queries.

Most Common: Firefox has DoH enabled by default.

Fix:

  • Firefox: Settings → Privacy & Security → "DNS over HTTPS" → set to "Off"
  • Chrome: Respects system DNS by default (no change needed)
  • Mobile apps: Usually no way to disable. Accept that Pi-hole won't affect these.

3. HTTPS-Based Ad Trackers with Certificate Pinning

Some enterprise tracking pixels, ad servers with pinned certificates

If an app has the IP address of an ad server cached internally or uses certificate pinning, it bypasses DNS entirely. The request goes directly to the IP without a DNS lookup.

Fix: None on the network level. These require app-level changes or blocking at a lower level (firewall rules by IP).

4. Local Network Ads (Bundled or Pre-Cached)

App ads that don't make DNS queries, local data files with ad content

Some apps have advertisements bundled into them or cached locally. They never make a DNS query, so Pi-hole has no way to interfere.

Examples: Mobile game ads, embedded webviews with local content.

Fix: Remove or downgrade the app, or pay for a premium version.

5. Streaming Platform Ads

Peacock, Paramount+, Hulu, Disney+, Amazon Prime, etc.

Most streaming services now serve ads from their own CDN infrastructure with domain names that are impossible to distinguish from content. Blocking them would break the service entirely. They've specifically designed their systems to avoid DNS-based blocking.

Fix: Pay for ad-free tiers or accept ads.

Pi-hole IS Effective For

Despite its limitations, Pi-hole is excellent at blocking:

  • Third-party banner ads and display networks (Google Ads, DoubleClick, AppNexus, etc.)
  • Analytics and tracking domains (Google Analytics, Facebook Pixel, etc.)
  • Telemetry from smart TVs, phones, and IoT devices
  • Malware and phishing domains
  • Ads on lesser-known websites and apps

The Bottom Line

Pi-hole reduces network-wide ad traffic by 20-40% in most homes, blocking ads on web browsing, IoT devices, and apps. It cannot block the 5 categories above without breaking services or requiring technological workarounds that are outside of DNS blocking.

If you're seeing ads on YouTube, Spotify, or streaming services and Pi-hole is confirmed working (Layer 4 verification passes), that's expected behavior, not a problem with your setup.
