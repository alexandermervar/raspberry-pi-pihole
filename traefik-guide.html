OVERVIEW

Chapter 1
What is Traefik?
A modern reverse proxy that auto-discovers your Docker containers and routes traffic to them — no manual config per service.
The traffic cop for your homelab
Traefik sits in front of all your services. Instead of remembering 192.168.4.100:9000 for Portainer and :8080 for something else, you visit portainer.homelab.local. Traefik figures out where to send the request automatically.

Install Traefik in Docker

Add labels to any service

Traefik routes it — done

Before Traefik, to access Pi-hole you had to know the exact IP and port. As you add more services — Portainer, Uptime Kuma, Grafana, your own web app — you accumulate a growing list of ports to memorize. Traefik solves this with automatic service discovery: it watches the Docker socket and routes based on hostname labels you attach to containers.

TUTORIAL: REVERSE PROXY 101 — What actually happens?

The problem without a reverse proxy

Each Docker container gets a port. Port 80 can only go to one app. So every service needs a unique port — :9000, :8080, :3000 — and you have to remember them all. Sharing your homelab with anyone else means sending them an IP + port combination. It doesn't scale.

What a reverse proxy does

A reverse proxy listens on ports 80 (and optionally 443) and acts as a dispatcher. When a request arrives for portainer.homelab.local, the proxy checks its routing table, sees "portainer → container:9000", and forwards the request internally. The client never sees the port. The container never needs to be on port 80.

Why Traefik specifically?

Nginx Proxy Manager and Caddy are alternatives, but Traefik has a killer feature for Docker homelabs: Docker label auto-discovery. Instead of editing a config file every time you add a service, you just add a label to the container. Traefik sees the label and starts routing immediately — no restart, no config file edit.

HOW IT WORKS

Chapter 2
Architecture
Understanding entrypoints, routers, and services before you touch a config file.
Browser request: http://portainer.homelab.local │ ▼ ┌─────────────────────────────────────┐ │ TRAEFIK CONTAINER │ │ │ │ Entrypoint :80 ──► Router │ │ Host(`portainer │ │ .homelab.local`)│ │ │ │ │ Service (lb) │ └──────────────────────────┼──────────┘ │ ▼ ┌─────────────────────────────────────┐ │ PORTAINER CONTAINER :9000 │ └─────────────────────────────────────┘

Traefik has three core concepts:

  • Entrypoints — the ports Traefik listens on (web = 80, websecure = 443)
  • Routers — rules that match incoming requests (e.g., Host(`portainer.homelab.local`))
  • Services — the backends that receive forwarded traffic (your containers)

When you add Docker labels to a container, Traefik automatically creates a router and service for it. The label traefik.http.routers.portainer.rule=Host(`portainer.homelab.local`) is all Traefik needs to start routing traffic to that container.

TUTORIAL: DOCKER NETWORK — Why traefik_proxy exists

The network requirement

Traefik needs to be on the same Docker network as the containers it routes to. Docker containers on different networks can't reach each other. We'll create a shared bridge network called traefik_proxy and attach both Traefik and every service you want routed to it.

Your existing setup

Pi-hole uses a macvlan network to get its own LAN IP. Traefik will live on a normal bridge network. Services that need both DNS (Pi-hole) and routing (Traefik) can be on multiple networks simultaneously in Docker Compose.

What about Pi-hole's macvlan?

Traefik will not be on the macvlan network. It doesn't need its own LAN IP. It just needs port 80 on the Pi's main IP, and a path to reach your containers via the traefik_proxy bridge network.

Step 2
Create the Proxy Network
A shared bridge network that Traefik and your services all join.
STEP 2 Create traefik_proxy network
⌨ SSH — on the Pi

Create a Docker bridge network that Traefik and all proxied services will share. We mark it external in compose files so it persists even when containers are stopped.

bash
docker network create traefik_proxy

bash

TUTORIAL: Bridge vs macvlan — which is which?

Bridge networks (traefik_proxy)

A Docker bridge network is an internal Docker virtual network. Containers on it can talk to each other by container name. Traffic goes out through the Pi's main network interface with NAT. This is what most Docker services use.

macvlan networks (lan_macvlan)

Pi-hole uses macvlan so it gets its own real LAN IP (e.g., 192.168.4.50). Other devices on your network see Pi-hole as if it's a separate physical machine. This is important for DNS — routers need to send DNS queries to a specific IP.

Traefik doesn't need macvlan

Traefik just needs port 80 on the Pi. The Pi already has a real LAN IP from Eero's DHCP reservation. Traefik binds to that IP on port 80, and Docker handles forwarding the traffic into the container via bridge networking.

STEP 3

Step 3
Traefik Static Configuration
The traefik.yml file — Traefik reads this once at startup.
STEP 3 Write traefik.yml
⌨ SSH — on the Pi

Create the static configuration file. This file tells Traefik which ports to listen on, how to discover services (Docker), and whether to show the dashboard.

bash — create the file
nano ~/homelab/traefik/config/traefik.yml

Paste the following:

~/homelab/traefik/config/traefik.yml
# Traefik Static Configuration
# This file is read once at startup.
# Dynamic config (routes) comes from Docker labels.

api:
  dashboard: true
  insecure:  true   # OK for homelab; disable if exposing externally

entryPoints:
  web:
    address: ":80"

providers:
  docker:
    endpoint:       "unix:///var/run/docker.sock"
    exposedByDefault: false    # only route containers with traefik.enable=true
    network:         "traefik_proxy"

log:
  level: "INFO"

Save with Ctrl+O → Enter → Ctrl+X.

TUTORIAL: What each section does

api.dashboard: true + insecure: true

Enables the Traefik web dashboard on port 8080. The insecure flag allows HTTP access without a password — fine for a local homelab, but if you ever expose Traefik to the internet, set this to false and add authentication middleware.

entryPoints.web

Traefik listens on port 80 for all incoming HTTP traffic. You can add a websecure entrypoint on port 443 later when you set up HTTPS with Let's Encrypt or mkcert.

providers.docker

This tells Traefik to watch the Docker socket for container events. exposedByDefault: false means Traefik ignores containers unless they explicitly have the label traefik.enable=true. This keeps things clean — only containers you opt in get routes.

providers.docker.network

When a container is on multiple networks, Traefik needs to know which one to use for routing. Setting network: traefik_proxy tells it to always use that network to reach containers.

TUTORIAL: Middleware — the missing piece of Traefik

Middleware sits between the router and the service — it can modify requests and responses before they reach your container. This is what makes Traefik powerful for adding security and routing logic without touching your application code.

Common homelab middleware

  1. BasicAuth — password-protect a service (useful for dashboard or internal tools)
  2. RedirectScheme — redirect HTTP → HTTPS automatically
  3. Headers — add security headers (HSTS, X-Frame-Options, etc.)
  4. RateLimit — throttle requests (protection against scanning)
  5. StripPrefix — strip a path prefix before forwarding (e.g., /app → /)

Example: Password-protect the Traefik dashboard

To add BasicAuth middleware to the dashboard, add these labels to the Traefik service in compose.yml:

- "traefik.http.middlewares.auth.basicauth.users=admin:$$apr1$$..."
- "traefik.http.routers.traefik-dash.middlewares=auth"

How middleware works

Middleware is defined by labels on any container (or in a file) and referenced by name from routers. In the example above, we define a middleware called auth, then reference it in the traefik-dash router. When a request matches that router, it flows through the auth middleware first.

This is what makes Traefik powerful for security — you can add authentication to ANY service without touching the service itself. Add labels, restart the container, done.

STEP 4

Step 4
Compose File
Define the Traefik container and its mounts.
STEP 4 Write compose.yml
⌨ SSH — on the Pi
bash
nano ~/homelab/traefik/compose.yml
~/homelab/traefik/compose.yml
services:
  traefik:
    image: traefik:v3   # v3 = latest stable v3 (currently v3.6.x) — pin to traefik:v3.6 if you want stability
    container_name: traefik
    restart: unless-stopped
    ports:
      - "80:80"        # HTTP entrypoint
      - "8080:8080"    # Traefik dashboard
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro
      - ./config/traefik.yml:/etc/traefik/traefik.yml:ro
    networks:
      - traefik_proxy
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.traefik-dash.rule=Host(`traefik.homelab.local`)"
      - "traefik.http.routers.traefik-dash.service=api@internal"
      - "traefik.http.services.traefik-dash.loadbalancer.server.port=8080"

networks:
  traefik_proxy:
    external: true   # use the network we created manually
⚠
Docker socket security: Mounting /var/run/docker.sock gives Traefik full control over Docker. This is standard practice for Traefik in a homelab, but on a public-facing server you'd use a socket proxy (e.g., tecnativa/docker-socket-proxy) to limit access.
TUTORIAL: The Docker socket — what it means and how to harden it

What is /var/run/docker.sock?

/var/run/docker.sock is the Unix socket Docker's daemon listens on. Any process that can write to this socket can interact with Docker — start, stop, and inspect containers. Traefik needs to read container information from this socket to discover routes.

What does :ro (read-only) actually mean?

Mounting it as :ro means Traefik can read container info but can't start/stop containers. This is safer than read-write, but it's not a full sandbox.

Why is this still a security concern?

Even read access to the Docker socket is sensitive. Any process that can read it can enumerate all containers on the host, their environment variables (which may contain secrets), and network configurations. This is risky if untrusted code runs on the same host.

For a homelab behind a home router, this is fine

The risk is from other processes running on the same Pi, not from the internet. If your homelab is only accessible on your home network and you control what runs on the Pi, read-only socket access is an acceptable trade-off for the convenience of Docker labels.

If you ever expose services to the internet

Consider tecnativa/docker-socket-proxy: a companion container that proxies the Docker socket and only exposes the specific API endpoints Traefik needs. This is a more robust security boundary between Traefik and the Docker daemon. The compose pattern looks like:

socket-proxy:
  image: tecnativa/docker-socket-proxy
  volumes:
    - /var/run/docker.sock:/var/run/docker.sock:ro
  environment:
    CONTAINERS: 1        # Allow container operations only
traefik:
  ...
  volumes:
    - /var/run/docker.sock:/var/run/docker.sock:ro  # Replace with socket-proxy
TUTORIAL: Understanding the compose labels

traefik.enable=true

Since we set exposedByDefault: false in traefik.yml, every container must explicitly opt in. This label is the opt-in.

traefik.http.routers.traefik-dash.rule

This creates a router named traefik-dash that matches requests for traefik.homelab.local. The router name (traefik-dash) is arbitrary — pick something descriptive.

traefik.http.routers.traefik-dash.service=api@internal

The Traefik dashboard is a special built-in service called api@internal. This label wires the router to that internal service rather than to a container port.

Why is restart: unless-stopped important?

Without it, Traefik won't start after a Pi reboot. With unless-stopped, Docker restarts Traefik (and all your services) automatically whenever the Pi boots — as long as you haven't manually stopped the container.

Step 7
Route Pi-hole Through Traefik
Add Traefik labels to your existing Pi-hole compose file so it gets a clean hostname.
STEP 7 Add Traefik labels to Pi-hole
⌨ SSH — on the Pi

Edit your Pi-hole compose.yml:

bash
nano ~/homelab/pihole/compose.yml

Add the highlighted sections to your Pi-hole service definition:

~/homelab/pihole/compose.yml — additions
services:
  pihole:
    image: pihole/pihole:latest
    container_name: pihole
    restart: unless-stopped
        # ... your existing config ...
    networks:
      - lan_macvlan      # existing — keeps Pi-hole's LAN IP for DNS
      - traefik_proxy    # NEW — allows Traefik to route the web UI
    labels:                # NEW block
      - "traefik.enable=true"
      - "traefik.http.routers.pihole.rule=Host(`pihole.homelab.local`)"
      - "traefik.http.routers.pihole.entrypoints=web"
      - "traefik.http.services.pihole.loadbalancer.server.port=80"
      - "traefik.docker.network=traefik_proxy"

networks:
  lan_macvlan:
    external: true
  traefik_proxy:    # NEW
    external: true

Then restart Pi-hole to apply the label changes:

bash — from the pihole directory
cd ~/homelab/pihole
docker compose up -d

Then add a Pi-hole DNS record: pihole.homelab.local → 192.168.4.100. Now visit http://pihole.homelab.local/admin.

ℹ
Pi-hole redirect note: Pi-hole's web server automatically redirects / to /admin, so http://pihole.homelab.local will take you straight to the admin panel.
TUTORIAL: The traefik.docker.network label explained

Why this label is necessary for Pi-hole

Pi-hole is on two networks: lan_macvlan (for DNS traffic) and traefik_proxy (for web UI routing). When Traefik sees a container on multiple networks, it picks one arbitrarily — often the wrong one. The traefik.docker.network=traefik_proxy label explicitly tells Traefik: "use this network to reach this container." Without it, Traefik might try to contact Pi-hole over the macvlan network and fail to connect.

Step 8 · Bonus
Self-Host a Web App
Deploy any containerized web app and route it through Traefik — this is what the whole setup makes possible.
STEP 8 Deploy a sample app (Nginx or your own)
⌨ SSH — on the Pi

The pattern for any web app is the same: put the service on traefik_proxy, add three labels, done. Here's a minimal example using Nginx to serve a static site — the same pattern applies to any Docker image.

bash
mkdir -p ~/homelab/mysite/html
echo '<h1>My Homelab Site</h1>' > ~/homelab/mysite/html/index.html
nano ~/homelab/mysite/compose.yml
~/homelab/mysite/compose.yml
services:
  mysite:
    image: nginx:alpine
    container_name: mysite
    restart: unless-stopped
    volumes:
      - ./html:/usr/share/nginx/html:ro
    networks:
      - traefik_proxy
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.mysite.rule=Host(`mysite.homelab.local`)"
      - "traefik.http.routers.mysite.entrypoints=web"
      - "traefik.http.services.mysite.loadbalancer.server.port=80"

networks:
  traefik_proxy:
    external: true
bash
cd ~/homelab/mysite
docker compose up -d

Add DNS record in Pi-hole: mysite.homelab.local → 192.168.4.100. Your site is live immediately — no Traefik restart needed. Traefik detected the new container via Docker socket and added the route automatically.

✓
This is the superpower: To add any future service, you just copy this pattern — three labels + the traefik_proxy network. Traefik handles the rest automatically, zero downtime, no config file edits.

STEP 9

Step 9 · Optional
Local HTTPS with mkcert
Upgrade from HTTP to HTTPS with locally-trusted certificates. Optional but useful for testing APIs that require HTTPS.
STEP 9 Set up HTTPS with mkcert or Let's Encrypt
💡
Why bother with HTTPS locally? Browsers show security warnings on self-signed certs, some modern APIs require HTTPS (geolocation, camera access, service workers), and it's good practice. Plus, setting this up now means your homelab infrastructure is production-ready.

Option A: mkcert (easiest for local HTTPS)

mkcert creates locally-trusted certificates that your browser will accept without warnings. Perfect for homelab internal services.

bash — on your Mac
brew install mkcert
mkcert -install
mkcert -cert-file ~/homelab/traefik/config/cert.pem -key-file ~/homelab/traefik/config/key.pem "*.homelab.local" "homelab.local"

Then copy the cert files to your Pi and update traefik.yml to use them:

traefik.yml — add websecure entrypoint and TLS
entryPoints:
  web:
    address: ":80"
  websecure:
    address: ":443"
    http:
      tls:
        certificates:
          - certFile: /etc/traefik/config/cert.pem
            keyFile: /etc/traefik/config/key.pem

Add 443:443 to ports in compose.yml, and use entrypoints=websecure in your router labels instead of web.

Option B: Let's Encrypt with DNS challenge (real certificates for internal services)

If you have a real domain and a DNS provider with an API (Cloudflare is easiest), you can use Let's Encrypt's DNS-01 challenge to get real trusted certificates for internal services. No port forwarding required — only your DNS provider needs to see the challenge.

This is worth it if you want real certs for production-grade homelab infrastructure:

traefik.yml — Let's Encrypt DNS challenge (Cloudflare)
certificatesResolvers:
  letsencrypt:
    acme:
      email: your-email@example.com
      storage: acme.json
      dnsChallenge:
        provider: cloudflare

Then reference the resolver in your router labels: traefik.http.routers.NAME.tls.certresolver=letsencrypt

Comparing the two approaches

ApproachCertificate sourceBrowser warning?Setup complexity
mkcertLocally-trusted (your computer)No — browser trusts itSimple
Let's Encrypt (DNS)Real trusted CANo — browser trusts itModerate (requires DNS API)

Recommendation: Start with mkcert for simplicity. If you need real certs and already own a domain, upgrade to Let's Encrypt DNS challenge.

TROUBLESHOOTING

Help
Common Issues
Things that go wrong and how to fix them.
✗ Port 80 is already in use — Traefik won't start

Find what's using port 80: sudo ss -tlnp | grep :80

Pi-hole's web server might be on port 80 if you mapped it that way. Check your Pi-hole compose file — if Pi-hole has "80:80" in ports, remove it. Pi-hole doesn't need a host port binding when Traefik is handling the routing. After removing the port mapping, run docker compose up -d in the pihole directory, then start Traefik again.

✗ Service not showing in Traefik dashboard

Most likely causes, in order:

  1. Missing traefik.enable=true label — this is the most common issue. Since exposedByDefault is false, every container must have this label.
  2. Container not on traefik_proxy network — check your compose file's networks: section and that you declared it as external.
  3. Compose file not reloaded — after adding labels, you must run docker compose up -d from the service directory to recreate the container with the new labels.
  4. Typo in label key — label names are case-sensitive. traefik.http.routers.NAME.rule must exactly match between router and service labels.
✗ 502 Bad Gateway when visiting a routed service

Traefik found the container but can't reach it. Causes:

  • Wrong port in label — the loadbalancer.server.port label must match the port the container listens on internally (not the host-mapped port). For Nginx this is 80, for Portainer it's 9000.
  • Container on wrong network — verify with docker inspect CONTAINER_NAME | grep -A 20 Networks. The container must appear in traefik_proxy.
  • Container crashed — run docker ps and check the status. If it's not running, check logs with docker logs CONTAINER_NAME.
✗ hostname.homelab.local doesn't resolve in browser

DNS issue — Traefik is working but the hostname isn't resolving. Check:

  1. Pi-hole DNS record exists — go to Pi-hole admin → Local DNS → DNS Records and verify the record is there.
  2. Your Mac is using Pi-hole as DNS — System Settings → Network → DNS. Pi-hole's IP should be the only entry.
  3. Quick test — run dig hostname.homelab.local @192.168.4.50 (using Pi-hole's IP). If it resolves there but not in the browser, your Mac isn't using Pi-hole as its DNS server.
  4. Workaround during testing — add an entry to /etc/hosts on your Mac: 192.168.4.100 myservice.homelab.local
✗ Traefik doesn't start after Pi reboot

Check that restart: unless-stopped is set in your compose file. Also verify the Docker service itself starts on boot: sudo systemctl is-enabled docker — it should say enabled. If not: sudo systemctl enable docker.

You can also check what happened on the last boot: docker logs traefik --since 24h

✗ Traefik routes HTTP to the right container but I want HTTPS

Add a websecure entrypoint on port 443 in your traefik.yml, add "443:443" to the ports section in your compose file, and use entrypoints=websecure in your router labels instead of entrypoints=web.

For trusted HTTPS on your local network, pair this with mkcert (see Step 9 — Local HTTPS). Without a valid certificate, browsers will show a security warning even though the connection is encrypted.

Quick test: If you only want to test HTTPS, you can accept the browser's security warning and proceed — the traffic is encrypted, just not from a trusted CA.

✗ I changed a label but nothing happened

Docker labels are only read when a container starts. After changing labels in your compose file, you must run docker compose up -d from that service's directory to recreate the container with the new labels.

The -d flag detaches it so it runs in the background. Traefik detects the new (or updated) container within seconds and updates its routing automatically — no need to restart Traefik itself.

Verify the change: Check the Traefik dashboard to see if the route appeared, or run docker logs traefik to see if there were any errors parsing the labels.
