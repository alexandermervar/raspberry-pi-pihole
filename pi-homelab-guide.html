Part 01
Concept Check
Read this before touching a terminal. Understanding the why prevents a lot of confusion later.

Containers vs. Virtual Machines

There are two ways to run isolated software: Virtual Machines and Containers.

A Virtual Machine emulates an entire computer — its own virtual CPU, RAM, storage, and a fully separate OS. Managed by a hypervisor (VMware, VirtualBox, Proxmox). Strong isolation, but expensive. A Pi 4 with 4–8GB RAM burns most of its headroom just keeping 1–2 VMs alive.

A Container shares the host's OS kernel but isolates everything above it — filesystem, processes, network, environment. Containers start in seconds, use a fraction of RAM, and a Pi 4 can run 10–15 comfortably.

Docker is the engine that builds and runs containers. Docker Compose lets you define containers in a plain YAML file so you can start, stop, and update them with single commands instead of memorizing long flags.

Virtual Machine · OS-level virtualization · Linux namespaces · Docker · YAML

Why Does Pi-hole Need Port 53?

Port 53 is the universal, standardized port for DNS traffic — hardcoded since 1983. Every device on your network sends DNS queries to port 53 on whatever DNS server your router's DHCP assigns. Pi-hole's job: sit on port 53, intercept every query, check it against blocklists, and either respond "blocked" or forward it to an upstream resolver like Cloudflare (1.1.1.1) or Google (8.8.8.8).

TUTORIAL — Set up SSH key authentication (passwordless + more secure)

Why bother?

Password auth works, but SSH keys are cryptographically stronger and eliminate the risk of brute-force password attacks. Once set up, you just type ssh homelab — no password prompt ever.

Step 1 — Generate a key pair on your Mac (if you don't have one)

TERMINAL — YOUR MAC
# Check if you already have a key
ls ~/.ssh/id_ed25519.pub

# If not, generate one (Ed25519 is modern and fast)
ssh-keygen -t ed25519 -C "amervar@mac"
# Accept default path, set a passphrase if you want an extra layer

Step 2 — Copy your public key to the Pi

TERMINAL — YOUR MAC
ssh-copy-id -i ~/.ssh/id_ed25519.pub amervar@homelab.local

This appends your public key to ~/.ssh/authorized_keys on the Pi. The Pi verifies you hold the private key (which never leaves your Mac) on every connection.

Step 3 — Test it

TERMINAL — YOUR MAC
ssh amervar@homelab.local
# Should connect without a password prompt

Optional: Disable password auth entirely on the Pi

SSH — PI
sudo nano /etc/ssh/sshd_config
# Find and change: PasswordAuthentication yes → no
sudo systemctl restart sshd
Only do this after confirming key auth works. If you lock yourself out with no key, the only recovery is reflashing the SD card.

STEP 05 Disable systemd-resolved
🔒 SSH — Pi via Mac Terminal
Don't skip this. It silently breaks most Pi-hole installs. If you already tried and got an error saying it wasn't running — that's fine, see the verify block below.

Modern Linux includes systemd-resolved, a background service that handles OS-level DNS lookups. It binds to port 53. Two services cannot own the same port — if systemd-resolved is running, Pi-hole can't start.

SSH — PI
# Stop it immediately
sudo systemctl stop systemd-resolved

# Prevent it restarting on boot
sudo systemctl disable systemd-resolved

# Remove the symlink it created
sudo rm /etc/resolv.conf

# Create a real file pointing at Cloudflare temporarily
# (Pi needs working DNS to download Docker in the next step)
echo "nameserver 1.1.1.1" | sudo tee /etc/resolv.conf

Part 03
Docker Networking for Pi-hole
The most conceptually dense section. Once it clicks, everything makes sense.

The Default Docker Network Problem

Docker creates a virtual bridge called docker0. Containers get private IPs in the 172.17.x.x range — not reachable from your LAN without port mapping. Port mapping works for web UIs, but DNS has a problem: Docker's NAT rewrites source IPs, so from Pi-hole's perspective every DNS query looks like it came from the Pi itself. Per-device tracking becomes useless.

Network bridge · NAT

The Solution: macvlan

macvlan is a Linux kernel feature that creates virtual network interfaces with their own MAC addresses, appearing to your LAN as independent physical devices. Docker uses this to give Pi-hole its own real LAN IP — completely separate from the Pi's IP.

From Eero's perspective: two devices are plugged in. Your Pi at 192.168.4.100 and Pi-hole at 192.168.4.2. DNS queries arrive at Pi-hole with real source IPs preserved. Device-level tracking works correctly.

One macvlan quirk: The Pi itself cannot communicate with the macvlan IP (kernel limitation). You access Pi-hole's web UI from your Mac, not from within SSH. On a headless server, this is never a real issue.
🔒 SSH — Pi via Mac Terminal

docker network create -d macvlan --subnet=192.168.4.0/24 --gateway=192.168.4.1 -o parent=eth0 lan_macvlan

~/HOMELAB/PIHOLE/DOCKER-COMPOSE.YML
version: "3.9"
    # ⚠ Deprecated in Compose v2 but harmless — you can safely remove this line entirely.

services:
  pihole:
    image: pihole/pihole:latest
        # Pulled from hub.docker.com. "latest" = newest release.
    # Pin to a version tag (e.g. "2024.07.0") for long-term stability.

    container_name: pihole
    hostname: pihole

    networks:
      lan_macvlan:
        ipv4_address: 192.168.4.2
                # Pi-hole's dedicated LAN IP — point Eero's DNS here.
        # Low numbers are safe — Eero's DHCP pool starts at ~.100.

    environment:
      TZ: "America/Chicago"            # your timezone
      WEBPASSWORD: "changeme"
            # Admin UI password. Pi-hole v6 may ignore this — set it post-deploy with:

      PIHOLE_DNS_: "1.1.1.1;8.8.8.8"
            # Upstream resolvers — Pi-hole forwards un-blocked queries here.
      # 1.1.1.1 = Cloudflare (fast, privacy-focused)
      # 8.8.8.8 = Google (reliable fallback)
      # 9.9.9.9 = Quad9 (malware blocking + privacy — worth swapping in)
      # ⚠ Pi-hole v6 (2024+) renamed this to FTLCONF_LOCAL_IPV4 — check your image tag.
      DNSMASQ_LISTENING: "all"        # accept queries on all interfaces

    volumes:
      - ./etc-pihole:/etc/pihole
      - ./etc-dnsmasq.d:/etc/dnsmasq.d
            # host_path:container_path
      # Pi-hole writes config/blocklists to /etc/pihole inside the container.
      # Mapping to real disk means data survives container recreation/updates.

    restart: unless-stopped
        # unless-stopped = restart on crash/reboot, not if you manually stopped it

    cap_add:
      - NET_ADMIN
            # Grants network admin rights without full root access.

networks:
  lan_macvlan:
    external: true
        # Network was created manually above — Compose won't try to manage it.

Point Your Eero at Pi-hole
The final step — activates Pi-hole for every device on your network.
📱 Eero App — on your phone
CRITICAL: Disable eero Secure and Ad Blocking FIRST. If you have eero Secure or eero Plus, these features intercept all DNS traffic at the network level, overriding any custom DNS setting you configure. Without disabling them, Pi-hole will not block ads on your network. Instructions below.

Step 1: Disable Advanced Security Features (if you have eero Secure or Plus)

  1. Open the eero app → tap the Menu icon (hamburger menu, top-left)
  2. Tap Network Settings → Advanced Settings
  3. Turn OFF Advanced Security
  4. Turn OFF Ad Blocking
  5. Go back to the main Network Settings screen

Step 2: Set Custom DNS

  1. From Network Settings, tap Advanced Settings → DNS → Custom DNS
  2. Enter Pi-hole's IP as primary: 192.168.4.2
  3. Leave secondary blank, or enter 1.1.1.1 as a fallback
  4. Confirm and save
Why leave secondary blank? A secondary DNS means devices silently bypass Pi-hole if it's unreachable. With no fallback, internet stops working — annoying, but immediate signal to go fix it.
TUTORIAL — How DNS fallback works

When does fallback actually trigger?

→
Pi-hole blocks a domain → responds instantly with "blocked." No fallback.
→
Pi-hole is completely down → device times out, falls to secondary. Internet works — Pi-hole bypassed silently.
→
Pi-hole is slow → devices wait. Fallback only triggers on true unreachable servers.
ConfigPi-hole goes downAd blocking
Primary onlyNo internet until fixedAlways enforced
Primary + SecondaryInternet works via fallbackBypassed silently

Once you add Uptime Kuma (on the project ideas list) to alert you when Pi-hole goes down, adding a secondary becomes more reasonable — you'll be notified before the fallback would ever matter.

Out of the box Pi-hole ships with one default blocklist. The real power comes from adding community-maintained lists. Each list is a plain text file of domains — Pi-hole merges them all and rebuilds its blocking database automatically.

Pi-hole v6 UI changes from v5:
  • "Adlists" is now called "Lists" (left sidebar)
  • "Whitelist" is now "Domains" → "Allowlist" (left sidebar)
  • "Query Log" is accessible from the left sidebar under "Logs"
To add blocklists: Click Lists in the left sidebar → paste a URL into the address field → click Add. After adding all your lists, click the Update Gravity button (also in the Lists page) to pull and compile them. Takes 30–60 seconds on a Pi 4.

Essential Docker Commands

Two types of commands — know which is which before running. docker compose commands require you to cd into the service folder first — they look for a docker-compose.yml in your current directory and fail with "no configuration file provided" if you're somewhere else. Plain docker commands work from anywhere — they target containers by name.

Plain docker — run from anywhere, target by container name:

SSH — PI — FROM ANY DIRECTORY
# See all running containers and their status
docker ps

# Watch live logs for a container (Ctrl+C to stop)
docker logs pihole -f

# Restart a container
docker restart pihole

# Open a shell inside a container
docker exec -it pihole bash

# Live CPU/RAM usage for all containers
docker stats

# Docker disk usage
docker system df

docker compose — must cd into the service folder first:

SSH — PI — MUST BE IN THE SERVICE FOLDER
# ⚠ cd into the folder FIRST — compose reads docker-compose.yml from here
cd ~/homelab/pihole        # or ~/homelab/portainer, etc.

# Start containers in the background
docker compose up -d

# Stop and remove containers (data in bind mounts is preserved)
docker compose down

# Pull latest image and restart
docker compose pull && docker compose up -d

# Live logs for this stack
docker compose logs -f

Troubleshooting
Common failure points and how to resolve them.
⚑Pi-hole container won't start
Run docker logs pihole. Most common cause: port 53 still occupied. Check: sudo systemctl status systemd-resolved. Also check: sudo ss -tlnp | grep :53
⚑Can't reach Pi-hole admin UI
Open https://192.168.4.2/admin from your Mac's browser, not from within the SSH session. The macvlan IP is only reachable from other LAN devices, not from the Pi itself. Pi-hole v6 uses HTTPS — expect a self-signed certificate warning; click "Advanced" → "Proceed anyway".
⚑I turned off my browser's ad blocker but still see ads on websites

Step 1: Check if eero Secure/Plus is blocking Pi-hole. If Advanced Security or Ad Blocking is enabled in your eero app, those features override custom DNS completely. Go back to the Eero section and disable them — without doing this, Pi-hole is being bypassed entirely.

Step 2: Verify Pi-hole is actually being used for DNS. From Terminal on your Mac, run nslookup google.com. The output should show Server: 192.168.4.2 — if it shows a different IP (like your ISP's DNS), your device isn't using Pi-hole at all.

Step 3: Understand what Pi-hole can and cannot block. Pi-hole only blocks DNS-based ads. It cannot block ads that are embedded in apps (YouTube, Spotify, TikTok) or first-party ads on pages like Reddit's promoted posts or Netflix's built-in ads. Those require browser extensions or app-level ad blockers.

Step 4: Verify Pi-hole is filtering this device. Run nslookup ads.doubleclick.net from Terminal. If it returns 0.0.0.0 or NXDOMAIN, Pi-hole IS blocking. If it returns a real IP address, Pi-hole is not filtering this device's DNS — go back to Step 2 and check your Mac's Wi-Fi settings.

Step 5: Check the Pi-hole dashboard query log. Open https://192.168.4.2/admin, click Logs in the left sidebar, and watch for queries. If the query counter shows 0 after waiting several minutes, your device is not sending DNS queries through Pi-hole at all — restart DHCP on your device (toggle Wi-Fi off/on for Mac/iPhone, ipconfig /renew for Windows).

⚑Devices still using old DNS after Eero change
DHCP leases hold for up to 24 hours. Mac: toggle Wi-Fi off/on or sudo killall -HUP mDNSResponder. iPhone: toggle Airplane Mode. Windows: ipconfig /renew
⚑SSH can't find homelab.local
mDNS occasionally fails. Use the IP directly: ssh amervar@192.168.4.100. Find the IP in the Eero app under Devices.
⚑Pi-hole is blocking something it shouldn't
Pi-hole UI → Domains → add to the allowlist. (In v6 the "Whitelist" is called the allowlist, found under Domains in the sidebar.) Common — popular blocklists occasionally catch legitimate domains.
⚑Pi completely unreachable over SSH
The one headless gotcha. Without a monitor, a reflash is your only recovery path. Prevention: don't edit /etc/ssh/ files directly. Your service data in ~/homelab/ bind-mount folders survives a reflash — reflashing only wipes the OS layer.
⚑"no configuration file provided: not found" on docker compose
You're not in the right directory. docker compose looks for a docker-compose.yml in whatever folder you're currently in. Fix: cd ~/homelab/pihole (or whichever service you're targeting), then re-run the command.
⚑YAML error on docker compose up
YAML requires spaces, not tabs. Open the file with nano and check for mixed indentation. If pasted from clipboard, tabs may have been introduced. Replace all tabs with 2 spaces.
⚑Forgot or lost the Pi-hole admin password
Reset it over SSH — no UI access needed. Pi-hole v6: docker exec -it pihole pihole setpassword — prompts for a new password interactively. Pi-hole v5: docker exec -it pihole pihole -a -p. Hit Enter twice at the prompt to clear it entirely (no password required to log in).
⚑Can't find Adlists in Pi-hole v6
Pi-hole v6 renamed it. Look for Lists in the left sidebar — that is the adlists manager. To whitelist a domain, go to Domains (also in the sidebar) and add it there.
⚑Pi-hole HTTPS shows a browser security warning
Expected behaviour when using a self-signed certificate — the encryption is real, but the browser doesn't recognise the issuer. Click "Advanced" → "Proceed anyway" and the browser remembers your choice. The permanent fix is a reverse proxy (Nginx Proxy Manager) with a proper certificate — that's Step 2 in the rollout order.
